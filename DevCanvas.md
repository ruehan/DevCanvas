# 개발자를 위한 AI 디자인 플랫폼 기획안

## 1. 제품 컨셉

이 플랫폼은 단순히 **“AI가 예쁜 화면을 만들어주는 툴”**이 아니라, 개발자가 실제로 사용할 수 있는 **UI 설계, 디자인 시스템, 상태별 화면, 프론트엔드 코드, 구현 가이드**까지 생성해주는 도구로 기획하는 것이 좋다.

핵심 포지션은 다음과 같다.

> **개발자가 바로 구현할 수 있는 UI 설계서, 디자인 시스템, 화면 상태, 프론트 코드까지 생성해주는 Dev-first AI Design Platform**

Google Stitch나 Claude Design처럼 자연어로 UI를 생성하는 흐름은 가져가되, 차별화 포인트는 **디자인 생성 자체**가 아니라 **개발 가능한 결과물**에 둔다.

---

## 2. 한 줄 설명

**요구사항을 입력하면 개발 가능한 UI 설계, 디자인 시스템, 상태별 화면, React/Tailwind 코드, 구현 가이드까지 만들어주는 개발자용 AI 디자인 플랫폼.**

---

## 3. 핵심 메시지

> **디자이너가 없어도 개발자가 혼자 서비스 화면을 설계하고 구현할 수 있게 한다.**

다만 “디자이너를 대체한다”는 메시지보다는 다음과 같이 표현하는 것이 더 현실적이고 설득력 있다.

> **개발자가 디자인 결정을 더 빠르게 내리고, 디자이너와 협업할 때도 훨씬 명확한 초안을 만든다.**

---

## 4. 왜 개발자에게 필요한가

개발자가 UI를 만들 때 실제로 막히는 부분은 단순히 색상이나 예쁜 디자인이 아니다. 실제로는 다음과 같은 문제에서 시간을 많이 쓴다.

1. 화면 구조를 어떻게 잡아야 하는지
2. 어떤 컴포넌트를 써야 하는지
3. 빈 상태, 로딩 상태, 에러 상태를 어떻게 처리해야 하는지
4. 모바일, 태블릿, 데스크톱 반응형을 어떻게 잡아야 하는지
5. 디자인 시스템 없이도 일관된 spacing, typography, color를 어떻게 유지할지
6. 기획서만 보고 실제 화면 흐름을 어떻게 나눌지
7. 만들어진 디자인을 코드로 옮길 때 얼마나 수정이 적을지
8. 관리자 페이지, SaaS 대시보드, 랜딩페이지, 온보딩 같은 반복 UI를 얼마나 빠르게 만들 수 있을지

따라서 플랫폼의 진짜 가치는 **예쁜 이미지 생성**이 아니라 **개발자가 구현 가능한 결정을 대신 정리해주는 것**이어야 한다.

---

## 5. 타겟 사용자

### 5.1 1차 타겟

**프론트엔드 개발자, 풀스택 개발자, 1인 개발자, 스타트업 개발자**

이 사용자들은 디자인 리소스가 부족하고, 빠르게 MVP나 관리자 페이지를 만들어야 하는 경우가 많다.

### 5.2 2차 타겟

**PM, 창업자, 기획자**

이들은 개발자에게 넘길 수 있는 UI 초안을 만들고 싶어 한다.

### 5.3 3차 타겟

**디자이너가 있는 팀**

이 경우에는 초안 생성보다 **디자인 시스템 적용, 변형안 생성, 개발 핸드오프**가 가치가 된다.

---

## 6. 핵심 차별화 방향

Claude Design이나 Google Stitch가 “디자인 생성” 중심이라면, 이 플랫폼은 개발자가 실제 구현에 사용할 수 있는 결과물에 집중해야 한다.

| 구분 | 기존 AI 디자인 툴 | 제안 플랫폼 |
|---|---|---|
| 주요 사용자 | 디자이너, PM, 창업자 | 개발자, 풀스택, 프론트엔드 |
| 결과물 | 예쁜 UI, 프로토타입 | 구현 가능한 UI 설계 + 코드 |
| 핵심 가치 | 빠른 시각화 | 빠른 구현 |
| 중요 기능 | 캔버스, 비주얼 편집 | 코드베이스 반영, 컴포넌트 매핑, 상태 설계 |
| 강점 | 디자인 탐색 | 개발 핸드오프, 코드 품질, 일관성 |

특히 AI 디자인 툴은 결과물이 비슷해지는 문제가 생기기 쉽다. 따라서 처음부터 **기본 AI 스타일을 내는 도구**가 아니라 **사용자의 코드베이스와 디자인 규칙을 반영하는 도구**로 설계하는 것이 중요하다.

## 7. 핵심 기능 기획

## A. 요구사항 → 화면 구조 생성

사용자가 다음처럼 입력한다고 가정한다.

> “B2B SaaS 관리자 페이지가 필요해. 고객 목록, 결제 상태, 계약 만료일, 검색, 필터, 상세 보기까지 있어야 해.”

이때 플랫폼은 단순히 예쁜 대시보드를 만드는 것이 아니라 다음을 생성해야 한다.

- 필요한 화면 목록
- 각 화면의 목적
- 사용자 흐름
- 컴포넌트 구조
- 데이터 테이블 컬럼
- 필터 조건
- 버튼 액션
- 빈 상태
- 로딩 상태
- 에러 상태
- 권한별 표시 여부
- 모바일 대응 방식

이 부분이 개발자에게 실제로 유용한 핵심 가치다.

---

## B. 화면 상태 자동 생성

개발자가 UI를 만들 때 자주 빠뜨리는 것이 상태 설계다.

예를 들어 고객 목록 화면이라면 다음 상태들이 필요하다.

- 기본 상태
- 검색 결과 없음
- 필터 적용 상태
- 로딩 상태
- API 에러 상태
- 권한 없음 상태
- 데이터 1개만 있는 상태
- 데이터가 너무 많은 상태
- 모바일 상태

이 기능은 제품 내에서 **State Matrix**라고 부를 수 있다.

| 상태 | 생성 결과 |
|---|---|
| Loading | skeleton UI |
| Empty | 안내 문구 + CTA |
| Error | 재시도 버튼 + 오류 메시지 |
| Permission denied | 권한 요청 안내 |
| Mobile | 카드형 리스트로 전환 |

이 기능은 일반적인 디자인 생성 툴보다 훨씬 개발자 친화적인 차별점이 될 수 있다.

---

## C. 디자인 시스템 자동 생성

프로젝트마다 다음과 같은 디자인 시스템 요소를 자동 생성하게 한다.

- 색상 토큰
- 폰트 스케일
- spacing scale
- radius
- shadow
- button variants
- input variants
- card variants
- table style
- modal style
- toast style
- empty state style

예시 출력:

```ts
export const tokens = {
  colors: {
    primary: "#2563EB",
    background: "#F8FAFC",
    text: "#0F172A",
  },
  radius: {
    sm: "6px",
    md: "10px",
    lg: "16px",
  },
  spacing: {
    xs: "4px",
    sm: "8px",
    md: "16px",
    lg: "24px",
  },
};
```

개발자 친화적인 포맷으로 다음 결과물을 지원하면 좋다.

- `design.json`
- `design.md`
- `tokens.ts`
- `tokens.css`
- `tailwind.config.ts`

## D. 코드베이스 기반 UI 생성

이 기능이 가장 강한 차별화 포인트다.

사용자가 프로젝트를 연결하면 AI가 아래를 읽는다.

- 사용 중인 프레임워크: React, Next.js, Vue 등
- 스타일 방식: Tailwind, CSS Modules, Styled Components 등
- 컴포넌트 라이브러리: shadcn/ui, MUI, Ant Design 등
- 기존 버튼, 카드, 모달 컴포넌트
- 라우팅 구조
- 폴더 구조
- API 응답 타입
- 기존 디자인 패턴

그다음 생성 결과는 사용자의 기존 프로젝트 방식에 맞춰야 한다.

나쁜 결과:

> “새로운 Button 컴포넌트를 만들어주세요.”

좋은 결과:

> “이미 `/components/ui/button.tsx`가 있으므로 해당 컴포넌트를 사용합니다. 고객 목록 화면은 `/app/customers/page.tsx`에 추가하고, 필터 컴포넌트는 `/components/customers/customer-filter.tsx`로 분리합니다.”

이렇게 해야 개발자가 진짜 쓸 이유가 생긴다.

## F. 디자인 리뷰 / UI 린터

개발자가 만든 화면을 업로드하거나 코드로 붙여 넣으면 AI가 리뷰해주는 기능이다.

리뷰 항목:

- spacing 일관성
- 색상 대비
- 버튼 우선순위
- 정보 위계
- 모바일 대응
- 테이블 가독성
- 접근성
- empty/loading/error 상태 누락
- 너무 많은 CTA
- 폼 검증 메시지 부족
- UX 흐름 어색함

이 기능은 생성 기능보다 더 자주 쓰일 가능성이 있다. 개발자는 “처음부터 만들어줘”보다 “내가 만든 거 괜찮은지 봐줘”를 더 자주 할 수 있기 때문이다.

## G. 프롬프트가 아니라 폼 기반 입력

개발자용 플랫폼이라면 채팅만 있어서는 부족하다.

입력 방식을 다음처럼 구성하면 결과가 더 안정적이다.

1. 만들고 싶은 화면 유형 선택
   - 랜딩페이지
   - 관리자 페이지
   - 대시보드
   - 설정 페이지
   - 로그인
   - 온보딩
   - 결제 페이지

2. 서비스 유형 선택
   - SaaS
   - 커머스
   - 내부툴
   - 모바일 앱
   - CRM
   - ERP

3. 사용자 역할 입력
   - 관리자
   - 일반 사용자
   - 매니저
   - 게스트

4. 주요 데이터 입력
   - 고객명
   - 결제 상태
   - 계약일
   - 담당자

5. 원하는 분위기 선택
   - 미니멀
   - B2B
   - 엔터프라이즈
   - 스타트업
   - 고급스러움
   - 친근함

   - React + Tailwind
   - Next.js + shadcn/ui
   - HTML/CSS

## 8. 주요 사용 시나리오

### 시나리오 1. 관리자 페이지 빠르게 만들기

개발자가 입력:

> “음식물 처리기 운영 관리 페이지를 만들고 싶어. 장비 목록, 상태, 온도, 가동 시간, 알림, 상세 페이지가 필요해.”

플랫폼 출력:

- 대시보드 화면
- 장비 목록 화면
- 장비 상세 화면
- 알림 설정 화면
- 상태별 UI
- React 코드
- API 타입 예시
- chart/table 컴포넌트
- 모바일 대응안

---

### 시나리오 2. 기획서 → UI 설계

사용자가 기획 문서를 붙여넣으면 플랫폼이 자동으로 다음을 생성한다.

- 기능 목록 추출
- 화면 목록 제안
- 사용자 플로우 생성
- IA 구조 생성
- 화면별 와이어프레임 생성
- 구현 우선순위 제안

---

### 시나리오 3. 기존 화면 개선

개발자가 현재 페이지 스크린샷 또는 코드를 업로드하면 플랫폼이 다음을 리뷰한다.

- 검색 필터가 너무 위계 없이 배치되어 있음
- 주요 CTA와 보조 CTA 구분이 약함
- 테이블 컬럼이 많아서 모바일에서 카드형 전환 필요
- 빈 상태가 없어서 신규 사용자 경험이 나쁨
- 상태 뱃지 색상이 의미 체계와 맞지 않음

그리고 수정된 버전을 바로 생성한다.

---

### 시나리오 4. 디자인 시스템 없는 팀을 위한 기본 세팅

사용자가 서비스 설명만 입력한다.

> “B2B 폐기물 관리 SaaS야. 신뢰감 있고 너무 화려하지 않은 톤이면 좋겠어.”

플랫폼 출력:

- 브랜드 톤
- 컬러 팔레트
- 타이포그래피
- 버튼 스타일
- 카드 스타일
- 테이블 스타일
- 상태 배지 스타일
- 차트 색상 규칙
- Tailwind config
- CSS variables

## 9. GLM 5 Turbo 활용 방향

GLM 5 Turbo는 단순 채팅 모델로 사용하는 것보다 여러 단계의 UI 생성 파이프라인을 구성하는 방식이 적합하다.

추천 구조는 다음과 같다.

### 1단계. Requirement Agent

사용자 입력을 분석해서 기능, 사용자, 화면, 데이터 구조를 뽑는다.

### 2단계. UX Planner Agent

사용자 흐름, 화면 구조, 상태 매트릭스를 생성한다.

### 3단계. Design System Agent

색상, 폰트, spacing, 컴포넌트 규칙을 생성한다.

### 4단계. UI Generator Agent

화면별 레이아웃과 컴포넌트 구성을 생성한다.

### 5단계. Code Generator Agent

React, Tailwind, shadcn/ui 기준 코드를 생성한다.

### 6단계. Review Agent

접근성, 반응형, 상태 누락, 디자인 일관성을 검사한다.

### 7단계. Handoff Agent

개발자가 바로 붙여넣을 수 있게 파일 구조, 설치 명령어, TODO, PR 설명을 생성한다.

향후에는 GLM 5V 계열 모델을 함께 사용해 스크린샷 분석이나 기존 UI 리디자인 기능으로 확장할 수 있다.

## 10. MVP 기능 범위

처음부터 Claude Design처럼 전체 디자인 캔버스를 만들려고 하면 범위가 너무 커진다. MVP는 더 좁게 가는 것이 좋다.

### MVP 목표

> **개발자가 프롬프트 하나로 Next.js + Tailwind 기반 화면 초안을 얻고, 상태별 UI와 코드까지 받을 수 있다.**

### MVP에 꼭 넣을 기능

1. 텍스트 프롬프트 입력
2. 화면 유형 선택
3. 디자인 톤 선택
4. UI 미리보기
5. React/Tailwind 코드 생성

## 12. 출력물 구조

생성 결과는 단순히 화면 하나가 아니라 다음과 같은 프로젝트 구조로 나와야 한다.

```txt
/generated
  /app
    /dashboard
      page.tsx
  /components
    dashboard-card.tsx
    status-badge.tsx
    data-table.tsx
    empty-state.tsx
  /lib
    mock-data.ts
    types.ts
  /styles
    tokens.css
  design.md
  implementation-guide.md
```

`implementation-guide.md`에는 다음 내용이 들어가면 좋다.

- 생성된 화면 설명
- 컴포넌트 구조
- API 연결 위치
- 수정해야 할 mock data
- 접근성 체크포인트
- 반응형 기준
- 남은 TODO

이렇게 하면 개발자는 “그냥 디자인 이미지”가 아니라 “바로 개발에 쓸 수 있는 결과물”이라고 느낄 수 있다.

## 13. 플랫폼의 킬러 기능

### 13.1 State Matrix Generator

화면별 로딩, 빈값, 에러, 권한 없음, 모바일 상태를 자동 생성한다.

### 13.2 Codebase-Aware Design

기존 프로젝트 구조와 컴포넌트를 읽고 그 방식에 맞춰 UI를 생성한다.

### 13.3 Design-to-Code Handoff

디자인 결과를 바로 React/Tailwind 코드와 파일 구조로 변환한다.

### 13.4 UI Review Agent

개발자가 만든 화면의 UX/UI 문제를 자동 리뷰한다.

### 13.5 Design System Lite

디자인 시스템이 없는 팀도 최소한의 토큰과 컴포넌트 규칙을 자동 생성한다.

초기 우선순위는 다음이 좋다.

1. State Matrix
2. Code Export
3. UI Review

## 14. 경쟁 제품 대비 포지셔닝

### 14.1 Claude Design과의 차이

Claude Design류 제품이 디자인, 프로토타입, 슬라이드, 마케팅 자료까지 폭넓은 시각 작업을 다룬다면, 이 플랫폼은 범위를 좁혀 **개발자가 실제 프론트엔드 구현에 쓰는 UI 설계와 코드 생성**에 집중한다.

### 14.2 Google Stitch와의 차이

Google Stitch류 제품이 자연어 기반 고충실도 UI 생성, 무한 캔버스, 디자인 시스템 파일, 인터랙티브 프로토타입을 강조한다면, 이 플랫폼은 다음에 집중한다.

- 내 프로젝트에 붙는 코드
- 상태별 UI
- 개발 규칙 반영
- 기존 컴포넌트 재사용
- 구현 가이드 자동 생성

### 14.3 v0, Bolt, Lovable류와의 차이

v0, Bolt, Lovable류는 앱 전체 생성이나 웹앱 빌드에 가깝다.

이 플랫폼은 앱 전체를 만드는 것보다 **UI 설계 품질과 개발 핸드오프 품질**에 집중한다.

## 18. 가장 중요한 제품 원칙

### 18.1 예쁜 것보다 구현 가능한 것이 우선

디자인 이미지가 아무리 예뻐도 개발자가 다시 다 만들어야 하면 가치가 낮다.

### 18.2 한 화면보다 전체 상태가 중요

실제 서비스는 기본 화면보다 empty, error, loading, permission 상태에서 품질 차이가 난다.

### 18.3 디자인 시스템을 자동으로 유지해야 한다

AI가 매번 다른 버튼, 다른 색상, 다른 spacing을 만들면 개발자는 사용하지 않는다.

### 18.4 결과물은 반드시 코드와 연결되어야 한다

“이런 느낌입니다”가 아니라 “이 파일에 이 컴포넌트를 넣으세요”까지 가야 한다.

### 18.5 AI 스타일을 줄여야 한다

AI가 자주 만드는 뻔한 SaaS 카드, 그라데이션, 과한 둥근 모서리, 의미 없는 아이콘을 줄이고 사용자의 도메인과 브랜드에 맞춰야 한다.

## 19. 추천 MVP 정의

처음 제품은 다음과 같이 정의하는 것이 좋다.

> **개발자가 서비스 설명을 입력하면 Next.js + Tailwind 기반의 화면 설계, 상태별 UI, 디자인 토큰, 구현 가능한 React 코드를 한 번에 생성해주는 AI UI 설계 도구.**

첫 버전에서는 특히 다음 영역에 집중하는 것이 좋다.

- 관리자 페이지
- SaaS 대시보드
- 내부툴 UI

이 영역은 개발자들이 반복적으로 만들고, 디자인 리소스가 부족하며, 생성된 코드의 실사용 가치가 가장 크다.

## 20. 최종 정리

이 플랫폼은 **AI 디자인 생성 툴**이라기보다 **AI UI 설계 및 구현 보조 플랫폼**으로 잡는 것이 좋다.

핵심은 다음 세 가지다.

1. **요구사항을 화면 구조와 상태별 UI로 바꿔준다.**
2. **디자인 시스템과 컴포넌트 규칙을 자동으로 만든다.**
3. **개발자가 바로 사용할 수 있는 코드와 구현 가이드를 제공한다.**

따라서 가장 강한 제품 문장은 다음과 같다.

> **디자인 감각이 부족한 개발자도, 기획만 입력하면 구현 가능한 UI와 코드를 바로 얻을 수 있는 개발자 중심 AI 디자인 플랫폼.**
