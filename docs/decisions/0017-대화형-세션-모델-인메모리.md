# 0017. 대화형 세션 모델 (B 방식, 인메모리)
- 상태: 채택
- 날짜: 2026-07-09

## 배경
MVP 는 무상태 1회성 `POST /generations`(폼 → 전체 결과)였다. Claude Code 식 스튜디오
(대화로 생성·반복 수정)를 지원하려면 (a) 세션/메시지 히스토리, (b) 진화하는 '현재 디자인'
상태, (c) 대화 엔드포인트가 필요하다.

## 결정
- **세션 모델**: `Session(id, messages[], current_result?)`. `Message(role, content, steps[])`.
- **영속성**: **인메모리 스토어**로 먼저 구현(프로세스 내 dict). Postgres 영속화는 이후 Phase.
  - 근거: 대화/편집 흐름을 빠르게 검증. 재시작 시 소실되지만 MVP-of-studio 에서 수용.
- **엔드포인트**:
  - `POST /sessions` — 세션 생성 → id
  - `POST /sessions/{id}/messages` `{prompt}` — 사용자 메시지 추가 + 처리 → 에이전트 메시지 + 갱신된 결과
  - `GET /sessions/{id}` — 현재 세션 상태
- **턴 의미론**:
  - 첫 메시지(`current_result` 없음) → 전체 파이프라인 실행(기존 7단계). 결과가 current_result.
  - 이후 메시지 → 편집 에이전트(current_result + instruction) → 수정된 current_result(ADR-0018).
- 첫 턴의 GenerationInput 은 프롬프트만으로 구성(나머지 기본값). 폼 필드는 향후 컨텍스트/설정으로 확장.

## 이유와 대안
- 인메모리: 속도·단순. DB 인프라 비용 없이 대화 흐름 검증. Postgres 는 저장/공유가 필요해지면 추가.
- 대안(처음부터 Postgres): 인프라 비용·속도 저하. 검증 전 과투자.
- 대화 엔드포인트 분리: 기존 `/generations`(1회성)는 호환성으로 유지하되, 스튜디오는 `/sessions` 사용.

## 영향
- `sessions/` 도메인 신설(schemas/store/service/router). 인메모리 스토어는 싱글톤(lru_cache 의존성).
- 진화하는 current_result 가 프론트 산출물 캔버스의 진실 원천이 됨.
- 재시작 시 데이터 소실 → 프론트에서 "세션 만료" 안내 필요.
- Postgres 이관 시: 스토어 구현만 교체(인터페이스 유지).
