# 0022. 편집 프롬프트에서 formal JSON 스키마 생략

- 상태: 채택
- 날짜: 2026-07-09

## 배경
ADR-0021 관측에서 편집 턴(apply_edit → GenerationResult 전체 재생성)이 무료 티어에서
429/timeout 으로 실패했고, 근본 원인은 재시도가 아니라 **요청 프롬프트 비대**로 지목됐다.

측정(fixture GenerationResult 기준):
- `_build_prompt` 가 넣는 formal JSON 스키마(`GenerationResult.model_json_schema()`) = 5762자.
- 현재 결과 덤프(`current_result`) = 2659자.
- **스키마가 payload 의 ~68%.** 실 결과가 커질수록 결과 덤프 비중이 오르지만, 스키마는
  내용과 무관하게 항상 통째로 실린다(고정 낭비).

## 결정
편집 턴은 **formal JSON 스키마를 프롬프트에서 생략**한다.
- LLM 포트 `generate(..., *, include_schema: bool = True)` 옵션 추가(기본 True = 기존 동작).
- `edit_agent.apply_edit` 는 `include_schema=False` 로 호출.
- 스키마 대신 지시로 대체: "컨텍스트의 기존 결과와 완전히 동일한 JSON 구조를 유지한 채
  지시 부분만 수정하라." `current_result` 자체가 대상 스키마의 완전한 인스턴스이므로 구조 예시 역할.

## 이유와 대안
- **중복 제거**: 편집은 이미 유효한 GenerationResult 인스턴스를 컨텍스트로 받는다. 그 인스턴스가
  키·중첩·타입을 그대로 보여주므로 formal 스키마는 정보 중복. 실측 67% 프롬프트 축소
  (8539자→2804자, ~2846→934 tok).
- **ADR-0018 유지**: 편집 = 전체 재생성 방식은 그대로. 프롬프트 표현만 축약(방식 변경 아님).
- **최초 생성(requirement 등)은 include_schema=True 유지**: 그 단계는 완전한 인스턴스 예시가
  없어 스키마가 구조 가이드로 필요. 관측상 requirement 는 스키마 포함으로 100% 성공.
- **대안(부분 패치/diff 반환)**: 결과의 일부만 수정해 반환 → 더 큰 절감이나 병합 로직·정합
  위험 큼. 스코프 확대 → 기각(필요 시 별도 ADR).
- **대안(프롬프트 캐싱)**: 고정 스키마를 캐시 입력으로. 프로바이더 캐시 의존 + 편집은 결과가
  매번 달라 캐시 효율 제한 → 스키마 생략이 더 단순·확실.

## 영향
- 포트 시그니처 확장(`include_schema` 키워드, 기본값으로 하위호환). Dummy/GLM 어댑터 모두 반영.
- 검증: 단위 테스트 — 편집이 `include_schema=False` 전달, 스키마 미포함 프롬프트 생성 확인.
  프롬프트 67% 축소 측정.
- **재관측(정직)**: 67% 축소 후에도 실 edit 은 z.ai `glm-4.7-flash` 무료 티어에서 429 지속.
  격리 테스트(12s 쿨다운 + 단건 + 재시도 내장)에서도 429 → **연타 아티팩트 아님, 프롬프트
  크기 문제도 아님**. 원인 = 무료 티어 레이트/쿼터 벽(누적 관측 호출로 소진 추정. requirement 는
  통과하나 더 무거운 edit 은 실패). ⇒ 실 edit 검증은 **유료 티어(또는 쿼터 리셋) 전제** — 후속.
- 즉 이 ADR 의 성과는 "프롬프트 67% 축소(비용·지연·쿼터 압박 감소)"로 확정하되, **무료 티어
  edit 통과는 미달성**. 단위 테스트로 로직 검증 완료.
- 결과 덤프 자체가 큰 경우(다수 화면·코드파일)는 여전히 클 수 있음 — 부분 패치/필드 선택 전송을
  후속 검토.
