# 0008. 디자인 시스템 에이전트는 규칙 기반(preset)
- 상태: 채택
- 날짜: 2026-07-01

## 배경
design_system_agent 는 색상·spacing·radius·typography·shadow 토큰을 생성한다.
이 영역은 LLM 보다 **결정적 규칙**(톤 → 프리셋 매핑)이 더 적합하다:
- 디자인 시스템은 일관성이 생명이므로 매번 LLM 이 다른 토큰을 내면 안 된다(기획안 18.3).
- 톤별 팔레트/spacing 스케일은 정형화된 도메인 지식이라 규칙으로 표현 가능.
- 결정적이어야 테스트가 가능하고, exporter(다중 포맷 변환) 검증도 확실하다.

## 결정
- design_system_agent 는 **규칙 기반**으로 동작한다. `pipeline/design/presets.py` 의
  톤 프리셋(B2B/minimal/enterprise/startup/friendly)에서 토큰을 생성한다.
- LLM 정제(브랜드 키워드 → 맞춤 색상 등)는 향후 확장. 현재는 llm.generate 를 사용하지 않는다.
- 토큰 → 다중 포맷(tokens.ts/tailwind.config.json/design.json/tokens.css/design.md) 변환은
  `pipeline/design/exporter.py` 가 담당한다. 순수 함수로 결정적.

## 이유와 대안
- 규칙 기반: 결정적·테스트 용이·일관성 보장. 도메인 지식이 코드로 명시돼 리뷰/조정 쉬움.
- 대안(LLM 기반): 유연하나 일관성·비용·비결정성 문제. MVP 에서 기각.
  향후 "사용자 브랜드 키워드 → 맞춤 팔레트" 정제만 LLM 에 맡기는 하이브리드 가능(별도 ADR).

## 영향
- design_system_agent 가 다른 에이전트와 달리 LLM 포트를 사용하지 않음 → 파이프라인에서
  LLM 의존이 하나 줄어 더미 없이도 동작하는 구간이 생김.
- 톤 프리셋이 도메인 지식의 단일 소스. 새 톤 추가는 presets.py 에 프리셋 추가로 끝.
- exporter 가 포맷별 산출을 보증하므로, 향후 핸드오프에 토큰 파일들을 쉽게 포함 가능.
