# 0009. 디자인 토큰 산출물을 코드에 연결하는 위치
- 상태: 채택
- 날짜: 2026-07-01

## 배경
ADR-0008 로 만든 exporter(토큰 → tokens.ts/tailwind.config.json/design.json/tokens.css/design.md)가
아무 곳에도 소비되지 않아 죽은 코드 상태였다. 파이프라인 결과(GenerationResult.code)에
토큰 파일들을 포함해 핸드오프/export 에서 바로 쓸 수 있게 해야 한다.

## 결정
- 토큰 파일 생성은 `pipeline/design/exporter.py` 의 순수 함수 `to_code_files(ds) -> list[CodeFile]`
  로 둔다(규칙 기반, 결정적).
- **연결 위치는 orchestrator** 이다. orchestrator 가 code_generator 의 산출(app/component 코드)과
  design_system 에서 변환한 토큰 파일을 합쳐 `GenerationResult.code` 를 구성한다.
- code_generator_agent 에게 design_system 을 넘기지 않는다 → 코드 생성 에이전트는 LLM 코드gen
  역할에 집중하고, 규칙 기반 토큰 export 는 별도 경로로 분리.

## 이유와 대안
- orchestrator 연결: 각 에이전트는 단일 산출 스키마를 유지, orchestrator 가 조립 책임(기존 역할과 일관).
- 대안(code_generator_agent 내부 병합): LLM 기반 에이전트에 규칙 기반 로직이 섞여 응집도 저하 → 기각.
- 대안(8번째 에이전트 추가): 7단계 모델(기획안 9장)을 깨님 → 기각. 향후 단계가 늘면 재검토.

## 영향
- GenerationResult.code 에 토큰 파일 5개가 항상 포함된다(앱 코드와 함께).
- exporter 가 실제 소비되어 테스트가 의미를 가짐.
- **경로 충돌 방지**: `RESERVED_TOKEN_PATHS`(5경로)를 예약. orchestrator 병합 시
  path 충돌이 발생하면 토큰 파일이 우선(대체)하고 경고 로깅.
- **후속 이관(handoff 갭)**: handoff_agent 는 병합 전 `code`(CodeGeneration)를 받으므로,
  현재 `file_tree`·`install_commands`·`guide_md` 가 토큰 파일을 반영하지 못한다
  (예: "tokens.css import", "tailwind theme.extend 적용" 단계 누락). 이 갭은 handoff 가
  code 에서 file_tree·install 단계를 유도하도록 정제할 때 함께 해결(별도 작업으로 추적).
- 핸드오프 정제 전까지는 산출물은 result.code 에서 직접 소비자가 취득 가능.
