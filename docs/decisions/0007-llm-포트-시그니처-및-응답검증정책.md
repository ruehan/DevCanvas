# 0007. LLM 어댑터 포트 시그니처 및 응답 검증 정책
- 상태: 채택
- 날짜: 2026-07-01 (2026-07-08 응답검증 정책 확정·glm-5.2 반영)

## 배경
ADR-0005는 시그니처를 `generate(prompt, schema) -> dict` 로 가정했으나, 2단계 구현 중
구조화 출력(Pydantic 검증)과 컨텍스트 전달이 필요해졌다. 또한 GLM 응답이 스키마를
위반할 때 재시도/수정 책임이 어댑터인지 에이전트인지 정하지 않으면 포트 전면 재설계 위험이 있었다.

## 결정
1. **시그니처(채택)**: `generate(schema: type[T], instruction: str, context: dict) -> T`
   - `T`는 Pydantic `BaseModel` 하위. 어댑터가 구조화 출력을 스키마로 검증해 인스턴스로 반환.
   - `instruction`은 한국어 지시문, `context`는 직렬화 가능한 누적 컨텍스트.
2. **응답 검증 정책(채택)**:
   - 어댑터가 GLM 응답(JSON)을 `schema.model_validate(data)` 로 검증한다(검증 책임 = 어댑터).
   - JSON 파싱/스키마 위반 시 **예외를 발생**시킨다(상위로 전파).
   - **MVP 에서는 재시도 없음**(결정적 파이프라인 우선). 향후 스키마 수정 프롬프트로 재시도하는
     정책을 도입할 때 별도 ADR로 확장.
3. **모델**: `glm-5.2`(settings.glm_model 기본값). GLM OpenAI 호환 chat completions 엔드포인트 사용.

## 이유와 대안
- 구조화 출력(Pydantic): 파이프라인 각 단계가 타입 안전하게 연결, 더미 fixture 도 동일 스키마 공유.
- 재시도 없음(MVP): 더미 단계에선 검증 실패가 없고, 실 GLM 응답 특성(위반 빈도)을 먼저 관측한 뒤
  재시도 정책을 정하는 것이 합리적. 어댑터가 예외를 던지면 파이프라인/엔드포인트에서 일관 처리.
- 대안 시그니처 `generate(prompt: str) -> str`(비구조화): 각 에이전트가 파싱 책임 → 중복/불일치 → 기각.

## 영향
- 어댑터 구현체(GLMAdapter)가 스키마 검증 책임. 더미는 fixture 매핑으로 우회.
- GLMAdapter 는 httpx 로 `/chat/completions` 호출, `response_format: json_object` 요청,
  응답 content(JSON)를 파싱해 Pydantic 검증. 키 미설정 시 즉시 예외.
- 3단계 이후 관측 결과에 따라 재시도 정책 ADR 추가 가능.
