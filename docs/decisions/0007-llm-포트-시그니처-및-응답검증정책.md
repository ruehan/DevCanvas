# 0007. LLM 어댑터 포트 시그니처 및 응답 검증 정책
- 상태: 채택 (시그니처) / 보류 (응답 검증·재시도 정책)
- 날짜: 2026-07-01

## 배경
ADR-0005는 시그니처를 `generate(prompt, schema) -> dict` 로 가정했으나, 2단계 구현 중
구조화 출력(Pydantic 검증)과 컨텍스트 전달이 필요해졌다. 또한 GLM 응답이 스키마를
위반할 때 재시도/수정 책임이 어댑터인지 에이전트인지 정하지 않으면 3단계 이후
포트 전면 재설계 위험이 있다.

## 결정
1. **시그니처(채택)**: `generate(schema: type[T], instruction: str, context: dict) -> T`
   - `T`는 Pydantic `BaseModel` 하위. 어댑터가 구조화 출력을 스키마로 검증해 인스턴스로 반환.
   - `instruction`은 한국어 지시문(프롬프트 초안), `context`는 직렬화 가능한 누적 컨텍스트.
2. **응답 검증/재시도 정책(보류)**: GLM 실구현체 프로토타입(스파이크) 이후 별도 ADR로 확정.
   - 후보: (a) 어댑터 내부 재시도 + 스키마 수정 프롬프트, (b) 에이전트가 검증 실패 시 재호출.
   - 3단계 진입 전 GLM 스파이크로 포트가 버티는지 먼저 확인한다.

## 이유와 대안
- 구조화 출력(Pydantic) 채택: 파이프라인 각 단계가 타입 안전하게 연결되고, 더미 fixture도
  동일 스키마를 공유해 검증 가능.
- 재시도 정책 보류: 더미 단계에서는 검증 실패가 없으므로 조기 결정은 근거 부족.
  GLM 응답 특성(스키마 위반 빈도)을 관측한 뒤 결정이 합리적.
- 대안 시그니처 `generate(prompt: str) -> str`(비구조화): 각 에이전트가 파싱 책임을 져서
  중복/불일치 위험 → 기각.

## 영향
- 어댑터 구현체는 스키마 검증 책임을 진다(현재 더미는 fixture 매핑으로 우회).
- 3단계에서 GLM 스파이크 결과에 따라 본 ADR의 "보류" 항목을 확정하거나 시그니처를 재검토.
