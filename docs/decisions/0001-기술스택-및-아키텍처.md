# 0001. 기술 스택 및 아키텍처 채택
- 상태: 채택
- 날짜: 2026-07-01

## 배경
DevCanvas는 생성 결과물 자체가 Next.js + Tailwind + shadcn/ui 코드다. 플랫폼 자체도
동일 스택으로 구현하면 생성 코드를 자기 자신에서 바로 렌더/검증할 수 있어 피드백 루프가 짧아진다.
백엔드는 제로원 메인 스택 기준이 필요하다.

## 결정
- 프론트: Next.js App Router + TypeScript + Tailwind CSS + shadcn/ui, FSD 구조.
- 백엔드: Python FastAPI, 도메인 모듈 구조(CONVENTIONS.md).
- 모노레포: `web/`(프론트), `api/`(백엔드).
- DB: PostgreSQL + Redis(작업 큐).
- 프리뷰: Sandpack.
- 검증: tsc + ESLint + Prettier + Lighthouse CI.

## 이유와 대안
- **프론트=출력 스택 동일**: 생성 코드를 그대로 임베드해 렌더/검증 가능. v0/Bolt 류가 유사 패턴 사용.
  - 대안 SaaS: 검증 불가, 도메인 UI 학습 어려움.
- **FastAPI**: 제로원 표준, 비동기 파이프라인에 적합.
  - 대안 Node.js: 단일 언어 장점이나 제로원 표준 위배.
- **Sandpack vs WebContainers**: WebContainers는 Next.js 서버 런타임까지 에뮬레이션하나 무겁고
  복잡. MVP는 client 컴포넌트 프리뷰로 충분(ADR-0004).

## 영향
- 한 팀이 TS/Python 두 스택을 다뤄야 함.
- 생성 코드 품질 기준이 명확: "이 레포에서 tsc 통과하는 코드".
