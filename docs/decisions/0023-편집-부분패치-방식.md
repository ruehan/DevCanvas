# 0023. 편집 = 부분 패치 방식 (ADR-0018 대체)

- 상태: 채택
- 날짜: 2026-07-09

## 배경
ADR-0018 은 편집 턴을 "전체 GenerationResult 재생성"으로 정했다(단순함 우선, 실 LLM 미관측).
실 GLM 관측에서 이 방식이 실패:
- 무료 티어(glm-4.7-flash): 편집 = 429(쿼터 벽).
- 유료 티어(glm-4.7-flashx): 429 사라졌으나 **timeout** — 전체 결과를 다시 생성하는 **출력**이
  거대해 120s 초과(재시도까지 ~6분 매달림).

핵심: 입력이 아니라 **출력(전체 결과 재생성)** 이 병목. "바뀌는 부분만 고치면 된다"(사용자 지적).

## 결정
편집을 **부분 패치**로 전환한다.
1. LLM 은 `GenerationResultPatch`(모든 최상위 필드 Optional)를 반환 — **변경되는 최상위 섹션만**.
   바뀌지 않는 섹션(특히 큰 `code`/`handoff`)은 생략.
2. `merge_patch`: 패치의 non-None 필드로 기존 결과를 **최상위 교체 병합** → GenerationResult 로
   재검증(결과는 항상 유효 스키마).
3. 프롬프트: `include_schema=False`(ADR-0022) 유지. 반환 형식은 `EDIT_INSTRUCTION` 이 지시 —
   "변경 섹션만, 단 포함하는 섹션은 **전체를 완전하게**(모든 필드 유지, 지시 부분만 변경)".

## 이유와 대안
- **출력 축소가 목적**: 편집은 보통 code/handoff 를 건드리지 않음 → 이 큰 섹션을 생략하면
  출력 토큰 급감 → timeout/비용 해소.
- **최상위 교체(딥머지 아님)**: 병합 로직 단순·예측가능. 섹션 내부는 LLM 이 완전체로 반환.
- **완전성 지시 필수(관측 교훈)**: 최초 "변경분만" 지시로는 LLM 이 섹션 내부 필드까지 부분만
  반환 → `ScreenSpec.purpose` 등 필수 필드 누락으로 스키마 검증 실패. "포함 섹션은 전체를
  완전히"로 강화하니 통과.
- **대안(JSON diff/RFC6902 경로 패치)**: 경로 지정 정확도 리스크 큼 → 기각.
- **대안(단계 라우팅, 접근 B)**: 6/7 단계가 규칙 기반이라 의도 분류 후 해당 단계만 재실행이
  더 싸다. 그러나 의도 분류 실패 리스크 → 자주 나오는 편집 유형에 대해 후속 최적화로 남김.

## 관측 결과(검증)
z.ai `glm-4.7-flashx`, 하네스 `--runs 1`:
- 전체 재생성: edit timeout(실패).
- 부분 패치(완전성 지시 전): edit = **스키마 오류**(섹션 내부 필드 누락) — 전송은 해결됨.
- 부분 패치(완전성 지시 후): **edit 1/1 성공**. requirement 3/3 유지.
⇒ 전송(timeout/429) → 스키마 → 성공으로 단계적 해소 확인.

## 영향
- `GenerationResultPatch` 신설(schemas). `edit_agent`: `apply_edit` = generate(patch) → merge_patch.
  `merge_patch` 공개(테스트/재사용). Dummy 는 requirement 만 바꾼 최소 패치 fixture.
- 세션 서비스 시그니처 불변(`apply_edit(current, instruction, llm) -> GenerationResult`).
- **일관성 한계(명시)**: 최상위 교체라 LLM 이 ux_plan 만 바꾸고 code 를 그대로 두면 code 가
  구식이 될 수 있음. 현재는 "영향 섹션을 LLM 이 함께 패치"에 의존. 결정적 재유도(접근 B)로
  일관성을 보장하는 것은 후속.
- ADR-0018 은 이 ADR 로 대체(전체 재생성 → 부분 패치).
