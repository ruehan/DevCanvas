# 0012. 코드 생성 에이전트는 규칙 기반 스켈레톤
- 상태: 채택
- 날짜: 2026-07-08

## 배경
code_generator_agent 는 React/TSX 코드 파일을 생성한다. 실제 동작하는 완성 코드는
LLM 이 내야 하지만, MVP 규칙 기반 단계에서는 **일관된 스켈레톤**(page.tsx + 컴포넌트
스텁 + types/mock)을 결정적으로 생성하는 것이 핸드오프 가치가 크다:
- design(0008)/ux(0010)/ui(0011)이 모두 규칙 기반 → 그 산출(UIGeneration)을 소비하는
  code_generator 도 동일 패턴이 자연스럽고, 결정적 입력으로 일관된 코드 보장.
- page.tsx 가 컴포넌트 트리를 렌더 순서대로 배치하고, 컴포넌트 스텁을 별도 파일로 제공해
  개발자가 "어디에 무를 구현할지" 바로 알 수 있음(기획안 18.4 결과물=코드 연결).
- 컴포넌트 구현 본체는 TODO 로 남기고, 향후 (a) LLM 정제 또는 (b) 검증 루프(tsc/eslint)로 채움.

## 결정
- code_generator_agent 는 **규칙 기반**으로 동작. `pipeline/code/generator.py` 의
  `build_code_generation(ui)` 가 UIGeneration 에서 CodeGeneration(files)을 도출.
- `pipeline/code/templates.py` 에 종류별 page.tsx 템플릿·컴포넌트 스텁·types/mock 생성기.
- 경로 규약: dashboard → `app/dashboard/page.tsx`, list → `app/<slug>/page.tsx`,
  detail → `app/<slug>/[id]/page.tsx`. slug는 화면명 한글 라벨 → 역매핑(고객→customer).
  미지정 라벨은 kind-index 폴백.
- 생성 코드는 ADR-0009 예약 토큰 경로(lib/tokens.ts, tailwind.config.json,
  tokens/design.json, styles/tokens.css, docs/design.md)와 충돌하지 않는다.
- LLM 정제·검증 루프는 향후 확장. 현재 llm.generate 를 사용하지 않는다.

## 이유와 대안
- 규칙 기반 스켈레톤: 결정적·일관·테스트 용이. page→컴포넌트 배치가 ui_generator 와 정합.
- 대안(LLM 코드gen): 품질 높으나 비결정·비용·일관성 위험. MVP 에서는 스켈레톤 먼저,
  향후 LLM 이 스텁 본체를 채우는 하이브리드(별도 ADR).

## 영향
- code_generator_agent 가 LLM 포트를 사용하지 않음 → 규칙 기반 파이프라인 구간 추가.
- GenerationResult.code 에 page.tsx(화면 수) + 컴포넌트 스텁(유니크 컴포넌트 수) +
  lib/types.ts + lib/mock-data.ts 가 포함. 토큰 파일(ADR-0009)과 병합.
- 컴포넌트 본체는 TODO — 실구현은 향후 LLM/검증 루프에서.
- page.tsx 가 import 하는 컴포넌트 경로(@/components/<kebab>)와 스텁 파일 경로가 일치해야
  (정합 규칙, 테스트로 고정 예정 — ui_generator 0011 교훈 반영).
