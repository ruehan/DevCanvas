# 0020. 스튜디오 재디자인 — 대화형 셸(design.html 기반)
- 상태: 채택
- 날짜: 2026-07-09

## 배경
design.html 이 스튜디오 뷰(`view-studio`)를 포함하도록 보강됨. Claude Code 식 대화형 셸:
툴바 + 2컬럼(좌=대화, 우=캔버스), 좌측은 empty/loading/result 3상태, 우측은 탭(미리보기/코드/
디자인시스템/상태매트릭스/리뷰/내보내기). 기존 `/studio`(폼+탭 1회성)를 이 셸로 교체하고
백엔드 `/sessions` API(ADR-0017/0018, 사이클 1)에 실연동.

## 결정
- `/studio` = 스튜디오 셸. 라우팅으로 `/`(랜딩) ↔ `/studio` 전환(design.html 의 `switchView` 대신
  Next 라우터 사용).
- **좌측 대화 패널**(features/studio): phase(empty/loading/ready) 상태기계. 빈 상태=예시 프롬프트
  칩 + 입력; 로딩=사용자 버블 + 에이전트 typing + 작업단계; 결과=대화 스레드(사용자/에이전트 버블
  + 파일 diff) + 후속 입력.
- **우측 캔버스**(widgets/studio-canvas): 빈/로딩(스켈레톤)/결과(기존 ResultViewer 탭 재사용,
  한국어 라벨). design.html 의 mock 미리보기 상태(default/loading/empty/error 고객목록)는
  Sandpack(ADR-0015)이 이미 상태 토글을 제공하므로 포팅하지 않음.
- **연동**: 첫 전송 시 `POST /sessions` 로 세션 생성 → `POST /sessions/{id}/messages` 로 메시지.
  백엔드가 첫 턴=전체 파이프라인, 이후=편집(ADR-0017/0018)을 분기.
- 상태 동기화: 대화 phase 와 캔버스 phase 는 단일 studio reducer 로 동기(단일 진실 원천).
- 디버그용 `state-btn`(빈/생성중/결과 수동 전환)은 데모용 → 실제 구현에선 제외(실제 API 상태로 구동).

## 이유와 대안
- design.html 포팅: 진실 원천(ADR-0019) 일관. 대화+캔버스 공존 = Claude Code 무드.
- 라우터 전환: SSR/히스토리/딥링크 유리. design.html 의 in-page hidden 토글보다 Next 에 맞음.
- ResultViewer 재사용: 탭/패널 자산 재활용, 중복 방지. 라벨만 한국어로.
- mock 미리보기 상태 생략: Sandpack 이 실시간 렌더+상태토글을 이미 제공(ADR-0015). mock 정적
  콘텐츠는 버려짐.

## 영향
- 기존 `app/studio/page.tsx`(폼+탭)는 새 셸로 완전 교체. `features/generate-ui`(폼)는 스튜디오에선
  미사용(랜딩/레거시용 보존 또는 차후 제거).
- 신규: features/studio(model reducer + conversation UI), widgets/studio-toolbar,
  widgets/studio-canvas, shared/api/sessions.
- Phase 동기로 빈/로딩/결과 전환이 대화·캔버스 양쪽에 일관됨.
- 백엔드 /sessions 가 실제 호출(더미 파이프라인 동작). GLM 키 설정 시 실 생성/편집.
