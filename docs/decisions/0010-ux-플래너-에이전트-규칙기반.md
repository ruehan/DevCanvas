# 0010. UX 플래너 에이전트는 규칙 기반
- 상태: 채택
- 날짜: 2026-07-01

## 배경
ux_planner_agent 는 화면 목록·컴포넌트·State Matrix(loading/empty/error/permission/mobile)를
생성한다. design_system(0008)과 같은 근거로 규칙 기반이 적합하다:
- 기획안 18.2 "한 화면보다 전체 상태가 중요" — 모든 화면에 대해 상태가 완결성 있게
  생성돼야 하므로 결정적 보장이 필수. LLM 은 상태를 누락하기 쉽다.
- 관리자/대시보드/내부툴(MVP 범위, ADR-0002) 화면은 정형 패턴(표 목록·상세·대시보드)이라
  템플릿화 가능.
- 결정적이어야 State Matrix(킬러 기능 13.1)의 품질이 보장되고 테스트 가능.

## 결정
- ux_planner_agent 는 **규칙 기반**으로 동작. `pipeline/ux/planner.py` 의
  `build_ux_plan(requirement, screen_type)` 이 RequirementSpec 에서 화면을 도출한다.
- 화면 종류(`ScreenKind`: list/detail/dashboard)를 도입해 종류별 컴포넌트·상태 템플릿 적용.
- LLM 정제(자연어 기획 → 맞춤 화면)는 향후 확장. 현재는 llm.generate 를 사용하지 않는다.

## 이유와 대안
- 규칙 기반: 모든 화면에 5상태가 빠짐없이 생성(완결성), 종류별 일관된 상태, 테스트 용이.
- 대안(LLM 기반): 유연하나 상태 누락·불일관·비결정성. MVP 에서 기각.
  향후 requirement 단의 자연어 해석만 LLM 에 맡기는 하이브리드 가능(별도 ADR).

## 영향
- ux_planner_agent 가 LLM 포트를 사용하지 않음 → 파이프라인에서 규칙 기반 구간 추가.
- 화면 수 = screen_type 주화면 + (data_entity × {list, detail}). 엔티티가 많아지면 화면이
  늘어나므로 cap 정책이 필요할 수 있음(초구현에서는 엔티티 수를 그대로 사용).
- requirement_agent(1단계)는 여전히 LLM/더미 — 그 data_entities 를 입력으로 소비.
