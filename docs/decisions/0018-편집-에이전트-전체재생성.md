# 0018. 편집 에이전트는 전체 결과 재생성(LLM)
- 상태: 채택
- 날짜: 2026-07-09

## 배경
대화형 스튜디오의 핵심은 "버튼을 둥글게", "고객 테이블에 계약일 추가" 식의 자연어 부분 수정.
규칙 기반 에이전트(ADR-0008~0016)는 입력의 순수 함수라 기존 결과를 대상으로 한 자연어
타겟 수정을 못 한다.

## 결정
- 편집 에이전트 `apply_edit(current_result, instruction, llm) -> GenerationResult` 를
  `pipeline/edit_agent.py` 에 둔다.
- **전체 결과 재생성 방식**: LLM(GLM 5.2) 에게 (현재 결과 + 사용자 지시) 를 주고,
  **수정된 전체 GenerationResult** 를 구조화 출력으로 받는다(스키마 검증, ADR-0007).
- 컨텍스트: `{current_result: <직렬화>, instruction: <사용자 지시>}`.
- 편집 턴의 에이전트 단계(노출용): `["요청 분석", "결과 수정"]`.

## 이유와 대안
- 전체 재생성: 구현 단순(스키마 하나), 포트(ADR-0007) 그대로 재사용, 더미/실제 동일 경로.
  트레이드오프: 토큰 비용 큼(결과가 클수록), LLM 이 맥락을 잃을 위험.
- 대안(영역별 부분 수정): 지시를 분류(토큰/스크린/코드)해 해당 에이전트만 수정. 정밀하나
  분류기+다중 에이전트 복잡. MVP 에서는 전체 재생성으로 출발, 관측 후 정제.
- 대안(매번 0부터 전체 파이프라인 재실행): 편집 의미 없음(이전 결과 무시). 기각.

## 영향
- 편집 턴은 LLM 의존(더미는 fixture 반환으로 wiring 검증). GLM 키 필요 시에만 실 편집.
- GenerationResult 전체를 LLM 컨텍스트로 직렬화 → 토큰 비용. 향후 (a) 요약/슬림 컨텍스트,
  (b) 영역별 수정으로 최적화 여지.
- 더미 환경에선 편집 결과가 fixture(고정)라 "실제로 바뀌었는지"는 검증 불가 → wiring/스키마
  검증에 만족. 실 GLM 으로 품질 관측 후 정책 보강.
