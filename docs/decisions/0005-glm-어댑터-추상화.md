# 0005. GLM 어댑터 추상화, 더미 구현으로 선행
- 상태: 부분 대체됨 (위치·시그니처·응답검증정책 → 0006, 0007)
- 날짜: 2026-07-01

> 갱신(2026-07-01): 본 ADR 의 "어댑터 위치(`adapters/glm.py`)" 와 "시그니처
> `generate(prompt, schema) -> dict`" 은 2단계 구현(0006, 0007)에서 변경됐다.
> - 위치: `pipeline/llm.py` (0006)
> - 시그니처: `generate(schema, instruction, context) -> T` (구조화 Pydantic 반환, 0007)
> 핵심 결정(추상화 + 더미 선행 + 키는 .env)은 유효하다.

## 배경
GLM 5 Turbo API 접근 정보(키/엔드포인트)는 확보됐으나 세부(레이트리밋·응답 스키마·비용)가
아직 미정이다. 이를 기다리면 파이프라인 설계가 지연된다.

## 결정
- `adapters/glm.py`에 **LLM 어댑터 인터페이스**를 정의(`generate(prompt, schema) -> dict`).
  > 갱신: 위치·시그니처는 0006/0007로 이동. 위 갱신 노트 참조.
- 기본 구현은 **더미/목업 응답**(고정 JSON)으로 전 파이프라인을 end-to-end 돌릴 수 있게 한다.
- 실제 GLM 호출 구현체는 키/엔드포인트 확정 시 `.env` 주입으로 교체.

## 이유와 대안
- 인터페이스만 고정되면 7단계 에이전트 로직을 API 의존 없이 TDD로 구현 가능.
- 더미로 파이프라인·검증 루프·프론트 연동을 모두 선행 검증 → API 연결은 마지막 단계.
- 대안(API 우선): 키 대기로 블록, 응답 스키마 불확실로 재작업 빈번.

## 영향
- 어댑터 경계가 명확해야 함(프롬프트 템플릿·응답 스키마 검증 포함).
- 향후 Claude/GPT 등 타 모델로 교체 용이(전략 패턴).
- GLM 확정 시 실구현체 + 스키마 검증 보강을 별도 커밋으로 진행.
