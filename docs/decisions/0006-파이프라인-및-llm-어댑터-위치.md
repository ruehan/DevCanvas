# 0006. 파이프라인 및 LLM 어댑터 위치
- 상태: 채택
- 날짜: 2026-07-01

## 배경
ADR-0005에서 GLM 어댑터 추상화를 결정했으나 백엔드 내 구체적 위치를 미뤘다. 2단계에서
7단계 에이전트 파이프라인과 LLM 어댑터를 구현하며 위치를 확정한다.

## 결정
- 파이프라인과 에이전트: `api/src/devcanvas_api/pipeline/`
  - `pipeline/schemas.py` — 단계별 입출력 Pydantic 모델
  - `pipeline/llm.py` — `LLMAdapter` 포트(Protocol) + `DummyLLMAdapter` + `GLMAdapter`(stub)
  - `pipeline/agents.py` — 7단계 에이전트(requirement, ux_planner, design_system, ui_generator, code_generator, review, handoff). 에이전트가 늘어나면 `pipeline/agents/` 디렉터리로 분리.
  - `pipeline/orchestrator.py` — 순차 실행 + 누적 컨텍스트
  - `pipeline/fixtures.py` — 더미 결정적 데이터
- API 진입: `generations` 도메인(`POST /generations`)에서 orchestrator 호출.
- 파이프라인은 백엔드 내 횡단 도메인이므로 `core/`에 두지 않고 독립 슬라이스로 둔다.
  FastAPI 도메인 구조(CONVENTIONS)의 router 얇게/service에 로직 원칙을 그대로 적용한다.

## 이유와 대안
- 파이프라인은 비즈니스 로직의 핵심이므로 도메인 모듈로 취급 → `core`(인프라 설정)와 분리.
- 어댑터를 pipeline 내부에 둬 파이프라인과 응집. `core/llm` 대안도 검토했으나,
  파이프라인만 사용하므로 순환 의존 위험 없이 응집도가 높은 쪽을 택했다.
- 추후 타 도메인이 LLM을 필요로 하면 `core/llm`으로 승격(이동) — 그 시점에 ADR 추가.

## 영향
- `generations` 도메인이 `pipeline`에 의존(단방향). `pipeline`은 `core`(설정)에만 의존.
- LLM 포트가 Protocol이므로 테스트는 `DummyLLMAdapter` 주입으로 동작(GLM 키 불필요).
